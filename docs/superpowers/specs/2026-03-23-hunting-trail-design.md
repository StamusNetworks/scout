# Hunting Trail — Design Spec

## Summary

Rename and extract the "auto investigation" feature into a standalone "Hunting Trail" use-case under `threats/compromises`. The hunting trail is host-centric (not incident-centric): it takes a host IP and a date range, queries five event sources, and renders chronologically grouped timeline cards.

Remove the incident expanded row from the incidents table entirely. Wire the hunting trail component into two new surfaces: the Impacted Entities table expanded row and the host details page.

## Motivation

The auto-investigation data is tied to a host IP, not a specific incident. Decoupling it from `ThreatStatus` makes the feature reusable across any surface that has a host context. The "Hunting Trail" name better reflects what the feature does — it shows the trail of events associated with a host across multiple data sources.

## Scope

### In scope

- New `hunting-trail` use-case under `threats/compromises/use-cases/`
- Refactored hook: `useHuntingTrail({ asset, startDate, endDate })` — no `ThreatStatus` dependency, no ±7 day padding
- Renamed components: `HuntingTrail` (main), `HuntingTrailCard` (per-group card)
- Renamed model: `hunting-trail.model.ts` (types stay the same, just moved)
- New "Hunting Trail" tab in Impacted Entities expanded row (between Threats and Host Insights, `doc` type only)
- New "Hunting Trail" tab in host details page (after Incidents)
- Remove `ExpandedRow` and expander column from incidents table
- Delete detection methods component and all old auto-investigation files

### Out of scope

- Changes to the hunting trail card rendering logic (NRD/sightings/file/lateral/hunting cards stay as-is)
- Changes to the aggregate-timeline-events utility (logic unchanged, just moved)
- Changes to events.mocks.ts or sightings.mocks.ts (stay in their current locations)

## Component API

### HuntingTrail (main component)

The component calls `useGlobalQueryParams` internally to get dates, matching the pattern of sibling components in the entities expanded row (`ThreatsTimeline`, `HostInsightsBlocks`, `AttackerInfrastructure`).

```tsx
interface HuntingTrailProps {
  asset: string;       // Host IP address
}

export const HuntingTrail = ({ asset }: HuntingTrailProps) => {
  const { start_date, end_date } = useGlobalQueryParams(['dates']);
  const { groups, isLoading, isError, isEmpty } = useHuntingTrail({
    asset,
    startDate: start_date,
    endDate: end_date,
  });
  // Renders loading skeleton / error / empty / ScrollArea of HuntingTrailCard
};
```

### useHuntingTrail hook

```tsx
interface UseHuntingTrailParams {
  asset: string;
  startDate: number;
  endDate: number;
}

export function useHuntingTrail({ asset, startDate, endDate }: UseHuntingTrailParams) {
  // 5 parallel queries — dates passed directly, no ±7 day padding (caller controls the window):
  // 1. NRD alerts — useGetEventsQuery (alerts_tail) with stamus.nrd* flowbits
  // 2. Sightings — useGetSightingEventsQuery (es_discovery_events) for discovery.asset
  // 3. File events — useGetEventsTailQuery (events_tail, NOT alerts_tail) for fileinfo/DGA flowbits
  // 4. Lateral movement — useGetEventsQuery (alerts_tail) for smb_lateral critical
  // 5. Hunting alerts — useGetEventsQuery (alerts_tail) for stamus_type:hunting
  //
  // Returns: { groups: TimelineGroup[], isLoading, isError, isEmpty }
}
```

### HuntingTrailCard

Same as current `IncidentAutoInvestigationCard` — takes a `TimelineGroup`, renders a color-coded table. No API changes, just renamed.

## Integration Points

### Impacted Entities expanded row

In `impacted-asstets-table.expanded-row.tsx`, for `doc` type, add a "Hunting Trail" tab. The full resulting tab order for `doc` is:

> Timeline, Attacker infrastructure, Threats, **Hunting Trail**, Host Insights

```tsx
<TabsTrigger value="hunting-trail">Hunting Trail</TabsTrigger>
// ...
<TabsContent value="hunting-trail">
  <HuntingTrail asset={entity.value} />
</TabsContent>
```

The entity's `value` field is the IP address. `HuntingTrail` calls `useGlobalQueryParams` internally for dates.

### Host details page

Add a new route file at `src/routes/_enterprise/hosts/$hostId/hunting-trail.tsx` that renders `<HuntingTrail asset={hostId} />`. The route file is minimal — no search schema or pagination needed since the component loads all results internally.

In the host layout route (`route.tsx`), add a "Hunting Trail" tab link after "Incidents" in the tab bar. The full resulting tab order is:

> Insights, Timeline, Incidents, **Hunting Trail**, Detection Methods, Outlier Events, Detection Events, Sightings, Beacons

No badge count for the Hunting Trail tab — unlike the other tabs which show event counts, the hunting trail aggregates multiple sources and a single count wouldn't be meaningful.
